# Reflective Translation

LLM 反思式翻译质量检查与修复，通过系统化的批判性审查和批量修复，将机器翻译提升至接近人工翻译的质量水平。

---

## 工作流程

### 阶段 1：初步翻译（如需要）

如果用户提供的是待翻译文本，先生成初稿翻译：

1. **保留技术术语中英对照**：首次出现时使用"中文（英文）"格式
2. **直译优先，准确第一**：不必过分追求流畅，优先确保含义准确
3. **标记不确定的翻译**：对可能有多种翻译的术语添加注释

输出：初稿翻译文本

---

### 阶段 2：批判性审查

对翻译文本进行系统化的质量检查，识别并分类问题。

#### 检查维度

**严重误译（Critical Issues）** - 阻塞理解
- 比喻/习惯用语失效
- 关键名词误译
- 动词/动作完全错误

**表达不当（Major Issues）** - 影响阅读体验
- 典型的机器直译模式
- 词序不符合中文习惯
- 用词不够精准

**术语不一致（Minor Issues）** - 影响专业性
- 技术术语翻译不统一
- 未采用行业标准翻译

**语境错误（Contextual Issues）** - 可能导致误解
- 专业术语在上下文中的具体含义
- 比喻性表达在特定领域中的含义

#### 详细检查清单

参考 `references/quality-checklist.md` 获取完整的检查清单和示例。

#### 常见错误模式

参考 `references/common-errors.md` 了解常见机器翻译错误模式及修复方法。

输出：质量检查报告（分级列出所有发现的问题）

---

### 阶段 3：批量修复

根据问题严重程度进行批量修复：

**修复优先级**：
1. **P0 严重问题**（必须修复）：比喻失效、核心概念误译、关键动词错误
2. **P1 中等问题**（应该修复）：生硬表达、不自然搭配、逻辑关系错误
3. **P2 轻微问题**（建议优化）：术语不一致、轻微语序问题

**批量修复技巧**：
- 一次性读取全文
- 按严重程度分批修复
- 使用 Edit 工具精确替换
- 保持术语一致性

参考 `references/terminology-guide.md` 确保术语翻译符合行业标准。

---

### 阶段 4：质量报告

生成详细的修复报告，包含：

**报告模板**：
```markdown
## 翻译质量检查报告

### 严重问题（必须修复）
- **行 X**："原译" → **问题**：[问题描述] → **修复**："修复后"

### 中等问题（影响阅读）
- **行 Y**："原译" → **问题**：[问题描述] → **修复**："修复后"

### 轻微问题（术语优化）
- **行 Z**："原译" → **问题**：[问题描述] → **修复**："修复后"

### 修复统计
- 严重问题：X 处
- 中等问题：Y 处
- 轻微问题：Z 处
- 总计：X+Y+Z 处

### 术语对照表
- [关键术语的翻译选择]
```

## 使用场景

### 适用场景

✅ **技术文档翻译**
- API 文档
- 开发指南
- 技术博客
- 产品文档

✅ **需要保持术语一致性的长文本**
- 系列文档
- 多章节教程
- 技术书籍

### 不适用场景

❌ **文学翻译**
- 需要人工创意和文学性
- 比喻和修辞需要艺术化处理

❌ **法律/医疗等专业文档**
- 需要专业领域的法律/医疗翻译
- 术语准确性要求极高

❌ **营销/广告文案**
- 需要文化本地化和创意改编
- 情感色彩和品牌调性

---

## 关键原则

### 1. 批判优先于生成
LLM 作为"批评者"比作为"创作者"更可靠。批判性审查比从头翻译更容易。

### 2. 系统化检查
不是随机检查，而是按照明确的检查维度逐项验证。

### 3. 分批处理
按严重程度分批修复，避免遗漏重要问题。

### 4. 术语统一性
首次出现建立术语表，全文保持一致。
