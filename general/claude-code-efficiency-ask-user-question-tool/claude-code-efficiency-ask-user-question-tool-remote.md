## 核心观点

**AI 编程工具的输出质量完全取决于你的输入质量——大多数人效率低下的根本原因是规划太烂，而不是工具不行。**

---

## 引言：一个反直觉的现象

2026 年初，硅谷发生了一件反直觉的事：

当所有人都在疯狂追捧 Claude Code、Cursor 这些 AI 编程工具时，Anthropic 内部却出现了一个诡异的现象——那些效率暴涨 70% 的工程师，和那些抱怨"AI 生成垃圾代码"的工程师，**用的竟然是同一个模型**。

Ross Mike（一位在 AI 编程领域摸爬滚打数月的实战派开发者）道破了真相：

> "如果你生成的是垃圾，那是因为你给它的就是垃圾。"

---

## 死亡陷阱：你以为在用 AI，其实在给 Anthropic 捐款

Ross 发现了一个残酷的真相：**大多数人对 Claude Code 的使用方式，本质上就是在烧钱**。

### 典型错误流程

想象这样一个场景：

1. 你打开 Claude Code，输入"帮我做一个 TikTok 视频生成应用"
2. Claude 问你几个基础问题，你随便选几个选项
3. 它开始疯狂输出代码
4. 生成的界面不是你想要的，功能逻辑有问题
5. 你开始来回修改，Token 哗哗地烧
6. 最后你得到了一个"能跑但不能用"的半成品

Ross 把这种现象称为**"捐款式开发"**：

> "如果你的计划很烂，那 Ralph 循环再强大也没用，你只是在给 Anthropic 捐钱。而且我觉得 Anthropic 已经够有钱了，不需要你的捐款。"

**核心矛盾**：AI 编程工具的输出质量，完全取决于你的输入质量。

就像你在跟一个初级工程师沟通，如果你只说"给我做个登录功能"，他怎么知道：
- 弹窗登录还是独立页面？
- 要不要记住密码？
- 要不要第三方登录？

这些细节你不说清楚，AI 就会替你做决定。而它的决定，往往不是你想要的。

## 转折：一个被忽视的神器

就在大家都在研究 MCP、Skills、各种插件的时候，Ross 发现了一个被严重低估的工具：

### Ask User Question Tool

这个工具的逻辑很简单：它会像面试官一样，**连续不断地问你问题，直到把你的需求问清楚为止**。

#### 对比演示

**传统规划模式**：
- Claude Code 问你两三个基础问题
- 然后就开始生成代码了
- 规划文档可能只有几百字
- 很多关键细节都是 AI 自己假设的

**启用 Ask User Question Tool 后**：
- "你的理想工作流程是什么？线性的还是迭代式的？"
- "如何处理 API 成本？硬性预算还是按需付费？"
- "数据库用什么？托管方案是什么？"
- "UI 风格是极简风还是仪表盘风？"
- "视频存储怎么处理？即时下载还是云存储？"
- ... 问题会一直问下去

Ross 分享道：

> "有些朋友用了这个工具后跟我抱怨，说问题太多太烦了。但这恰恰是它的价值所在。"

**价值**：
- 在规划阶段多花 10 分钟
- 能在开发阶段省下几个小时的返工时间
- 更能省下大量的 Token 消耗
- 逼着你去思考那些你原本没想清楚的问题

## 三个反直觉的建议

基于数月的实战经验，Ross 给出了三个听起来反直觉、但经过验证的建议：

### 1. 如果你还没有部署过任何能用的应用，不要碰 Ralph

这听起来很奇怪，Ralph 不是自动化神器吗？为什么不用？

**Ross 的比喻**：

> "这就像你还不会开车，就直接买了辆特斯拉想用自动驾驶。理论上很酷，但你连方向盘都没摸过，怎么知道什么时候该接管？"

**建议**：
- 先手动构建每一个功能
- 手动测试每一个功能
- 培养对产品开发的"感觉"
- 学会如何写清晰的需求
- 学会如何判断功能是否达标
- 学会如何发现潜在问题

**只有当你能手动从零到一完整交付一个应用后，再去用 Ralph 这样的自动化工具。否则，自动化只会让你更快地走向错误的方向。**

### 2. 上下文管理比你想象的重要得多

Claude 的 Opus 4.5 模型有 20 万 Token 的上下文窗口，听起来很大。但 Ross 建议：

> **当上下文使用超过 50% 时，就该开启新会话了。**

**为什么？**

因为模型在上下文过载时会"失忆"。就像一个人被灌输了太多信息，开始记不清哪些是重点，哪些是次要的。

很多人抱怨"一开始生成得挺好，后来越来越差"，根本原因就是上下文塞得太满了。

### 3. 不要过度迷恋工具和插件

MCP、Skills、各种 Markdown 配置文件，这些东西都很酷。但 Ross 的观点很明确：

> **"这些东西不是你产品失败的原因。大多数时候，是你的计划太烂了。"**

他见过太多人花大量时间研究各种工具配置，却不愿意花时间好好写一份清晰的需求文档。

这就像一个厨师把所有精力都花在研究各种高级厨具上，却不愿意学习基本的烹饪技巧。

**工具是放大器，它会放大你的能力，也会放大你的问题。如果你的基础不扎实，再好的工具也救不了你。**

## 输入决定输出：一个被验证的铁律

Ross 的核心观点可以浓缩成一句话：

> **在 AI 编程时代，输入质量决定输出质量。**

这不是什么新鲜理论，但在 AI 编程领域，这个铁律被放大了无数倍。

### 传统开发 vs AI 开发

**传统编程**：
- 如果需求不清晰，工程师会来回跟你确认
- 这个过程虽然痛苦，但至少能保证最终产品不会偏离太远

**AI 开发**：
- AI 会基于你给的信息，做出它认为"最合理"的假设
- 然后一路狂奔
- 等你发现方向错了，可能已经烧掉了几千个 Token，生成了几千行代码

这就是为什么 Ross 如此强调规划的重要性。

### 好的规划文档标准

**一份好的规划文档，应该详细到什么程度？**

Ross 的标准是：

> **如果你把这份文档交给一个初级工程师，他应该能不问任何问题就开始动手。**

每一个功能的呈现方式、每一个交互的细节、每一个技术选型的理由，都应该写清楚。

这听起来很费时间？确实。但这个时间花得值。因为它会在后续的开发过程中，为你节省 **10 倍甚至 100 倍**的时间和成本。

---

## 但这不是万能药

Ross 很诚实地指出了这套方法的局限性：

### 1. 这套方法对初学者不友好

Ask User Question Tool 会问很多技术问题，如果你完全不懂技术，可能连问题都看不懂。

**Ross 的建议**：把不懂的问题复制到 ChatGPT 或 Claude 里，让它帮你解释和决策。

### 2. 过度规划也有风险

如果你花了两天时间写规划文档，最后发现这个想法根本不可行，那这两天就白费了。

**Ross 的建议**：先做一个最小可行性验证，确认方向对了，再投入大量时间做详细规划。

### 3. 规划再好，也替代不了品味和审美

2026 年，克隆一个功能已经很容易了，但做出一个让人"停下滚动"的产品，依然需要人的创意和品味。

**案例**：有人做了一个基于情绪的跑步应用，你告诉它你现在的情绪状态，它会生成一条个性化的跑步路线。这个想法本身就很有创意，而且在动画、配色、交互细节上都经过了精心设计。

**这种产品，AI 能帮你实现，但创意本身必须来自人。**

## 结语：你是想做打字员，还是想做指挥官？

Ross 在分享的最后说了一句很扎心的话：

> **"软件开发正在变得简单，但软件工程依然很难。"**

什么意思？

- **写代码、实现功能**，这些"开发"层面的工作，AI 已经能做得很好了
- **架构设计、用户体验、产品品味**，这些"工程"层面的工作，依然需要人的深度思考

2026 年，每个人都能用 AI 克隆一个价值 60 亿美元的软件。但如果大家都能做到，那你的软件凭什么脱颖而出？

答案是：**大胆和品味**。

- **大胆**：敢于尝试别人没做过的东西，敢于挑战常规，敢于在不确定中前行
- **品味**：知道什么是好的设计，什么是糟糕的体验，什么细节值得打磨，什么功能该果断砍掉

**这两样东西，AI 给不了你。**
